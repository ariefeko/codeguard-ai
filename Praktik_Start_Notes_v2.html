Memory Optimization

Atur Swap — Lakukan Sekali

Kenapa Perlu Swap?

Saat Ollama + Docker + VSCode + browser jalan bersamaan, RAM 16GB bisa mepet. Swap adalah "RAM cadangan" di SSD — lebih lambat dari RAM tapi mencegah sistem crash atau OOM (Out of Memory) killer yang tiba-tiba matikan proses.

bash — Setup 12GB Swap (jalankan sekali)
# Cek swap yang ada sekarang
swapon --show
free -h

# Buat file swap 12GB
sudo fallocate -l 12G /swapfile
sudo chmod 600 /swapfile
sudo mkswap /swapfile
sudo swapon /swapfile

# Supaya aktif setelah reboot — tambah ke /etc/fstab
echo '/swapfile none swap sw 0 0' | sudo tee -a /etc/fstab

# Verifikasi
free -h
# Harusnya sekarang ada Swap: 12.0G
bash — Tuning Swappiness (opsional tapi bagus)
# Default swappiness = 60 (agresif pakai swap)
# Turunkan ke 10 — pakai RAM semaksimal mungkin dulu
echo 10 | sudo tee /proc/sys/vm/swappiness

# Permanen:
echo 'vm.swappiness=10' | sudo tee -a /etc/sysctl.conf
Resource Monitoring

Pantau Resource — Habit Wajib

Biasakan buka monitor resource setiap kali mulai sesi AI. Ini akan membantu kamu mendeteksi bottleneck dan memahami kenapa kadang lambat.

📊 Tool Monitoring
bash — Install tools (sekali)

# nvtop / gpustat optional untuk non-NVIDIA   # pantau GPU usage
bash — Monitoring commands
# RAM + CPU real-time (paling sering dipakai)
htop

# Ringkasan cepat RAM
free -h

# Model Ollama yang sedang loaded di memory
ollama ps

# Berapa RAM yang dipakai Ollama
ollama ps --json

# Docker container yang jalan + resource usage
docker stats

# Disk usage per folder
du -sh ~/.ollama/models/*
du -sh ~/projects/*
💡 Tanda-tanda RAM Kritis
  • Ollama jadi sangat lambat tiba-tiba → RAM mepet, model swap ke disk
  • Sistem terasa lag saat inference → tutup browser tab dan aplikasi lain
  • Docker container mati sendiri → OOM killer, perlu lebih banyak swap
  • free -h menunjukkan available < 1GB → bahaya, tutup aplikasi
🟢 Ollama Service Check

Di Ubuntu, Ollama sering otomatis berjalan sebagai service setelah install.

bash — cek status
systemctl status ollama

Jika status active (running), tidak perlu menjalankan ollama serve manual setiap sesi.

Practical Habit

Rutinitas Sebelum Sesi AI

✅ Checklist Sebelum Mulai
  • Tutup tab browser yang tidak perlu (setiap tab = ~100MB)
  • Cek apakah Ollama service sudah aktif: systemctl status ollama
  • Cek RAM kosong: free -h — pastikan > 4GB available
  • Buat git branch baru sebelum AI mulai kerja
  • Commit baseline: git commit -m "baseline"
  • Buka htop di terminal terpisah untuk monitoring
🧹 Cleanup Setelah Sesi
  • Stop model dari memory jika tidak dipakai: ollama stop qwen2.5-coder:7b
  • Stop Docker containers yang tidak perlu: docker compose down
  • Review dan commit atau discard perubahan AI: git status
Support Notes Tambahan

Hal-hal yang Perlu Diketahui

🌡️ Thermal & Battery
Laptop + Inference = Panas

CPU inference itu berat. ThinkPad T14 punya cooling yang decent, tapi saat inference model 7B dalam waktu lama, CPU bisa throttle karena panas — yang membuat inference makin lambat.

bash — Monitor suhu CPU
sudo apt install lm-sensors -y
sensors        # lihat suhu semua komponen
watch sensors  # update setiap 2 detik

Tips: pakai charger saat inference — laptop mode baterai biasanya throttle CPU untuk hemat daya, membuat inference 2–3x lebih lambat.

⚡ Quick Performance Tips
Cara Percepat Inference di CPU
  • Set num_ctx sekecil yang cukup — context besar = lebih lambat. Untuk coding task biasanya 2048–4096 cukup
  • Model tetap loaded di memory setelah pertama dipanggil — request kedua dan seterusnya lebih cepat
  • Tutup semua app berat: Slack, Zoom, browser dengan banyak tab
  • Gunakan charger — jangan inference di mode baterai
  • Ollama secara default pakai semua CPU thread — tidak perlu konfigurasi tambahan
bash — Set context kecil untuk coding task
# Lewat Modelfile custom
cat > Modelfile << 'EOF'
FROM qwen2.5-coder:7b
PARAMETER num_ctx 4096
PARAMETER num_thread 10
EOF

ollama create qwen-fast -f Modelfile
ollama run qwen-fast
